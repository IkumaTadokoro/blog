## 前提

- [ssh](http://d.hatena.ne.jp/keyword/ssh)クライアント：[macOS](http://d.hatena.ne.jp/keyword/macOS) Catalina（10.15.6）
- 接続先：さくら[VPS](http://d.hatena.ne.jp/keyword/VPS)に構築したDebian10.4
- 以前は公開鍵認証で問題なく接続できていたが、突然`Permission denied (public key)`により接続できなくなった。
- 公開鍵認証については、[ssh接続を鍵認証で行う](http://www.tooyama.org/ssh-key.html)を参考に実施済み。つまり
  - client\_[rsa](http://d.hatena.ne.jp/keyword/rsa)というキーペアを作成
  - クライアントのキーチェーンには[秘密鍵](http://d.hatena.ne.jp/keyword/%C8%EB%CC%A9%B8%B0)を登録済（`ssh-add -K ~/.ssh/client_rsa`）
  - [Debian](http://d.hatena.ne.jp/keyword/Debian)側にも公開鍵を登録済
  - [Debian](http://d.hatena.ne.jp/keyword/Debian)側の[sshd](http://d.hatena.ne.jp/keyword/sshd)\_configは以下の設定がしてある

```
PasswordAuthentication  noChallengeResponseAuthentication  noUsePAM  yes
```

## 思い出せ[Debian](http://d.hatena.ne.jp/keyword/Debian)編

### 覚えてるか？俺たち、昔はパスワードで[ssh](http://d.hatena.ne.jp/keyword/ssh)接続したりしてさ...

- 確認したいこと：原因が公開鍵認証によるものなのかの切り分け
- 確認方法：[Debian](http://d.hatena.ne.jp/keyword/Debian)側の[sshd](http://d.hatena.ne.jp/keyword/sshd)\_configで`PasswordAuthentication yes`に設定変更する。
- 確認結果：パスワード認証では問題なく[ssh](http://d.hatena.ne.jp/keyword/ssh)接続できる。

[Debian](http://d.hatena.ne.jp/keyword/Debian)「今もパスワード認証できるで」

### 覚えてるか？この鍵。半分、お前に渡してさ...

- 確認したいこと：公開鍵がちゃんとサーバ側に格納されているかどうかを確認
- 確認方法：[mac](http://d.hatena.ne.jp/keyword/mac)で作成した公開鍵を確認し、[debian](http://d.hatena.ne.jp/keyword/debian)側のフォルダに配置されているものと比較する。
  - 具体的には以下の流れでクライアント、サーバでのcat結果を比較。

```
# クライアント（mac）にキーペアが作成されていることを確認$ ls ~/.ssh$ cat ~/.ssh/client_rsa.pub# サーバ（debian）に公開鍵が配置されていることを確認$ ls ~/.ssh$ cat ~/.ssh/authorized_keys
```

- 確認結果：作成した公開鍵がサーバ側に正しく格納されている。

[Debian](http://d.hatena.ne.jp/keyword/Debian)「今もちゃんと持ってるで、覚えとるで」

### なあ[Debian](http://d.hatena.ne.jp/keyword/Debian)、閉ざした心を開いてくれよ...

- 確認したいこと：エラーメッセージにある`Permission Denied`が本当に[パーミッション](http://d.hatena.ne.jp/keyword/%A5%D1%A1%BC%A5%DF%A5%C3%A5%B7%A5%E7%A5%F3)が原因か確認する。
- 確認方法：クライアント、サーバ側それぞれの.[ssh](http://d.hatena.ne.jp/keyword/ssh)[ディレクト](http://d.hatena.ne.jp/keyword/%A5%C7%A5%A3%A5%EC%A5%AF%A5%C8)リの[パーミッション](http://d.hatena.ne.jp/keyword/%A5%D1%A1%BC%A5%DF%A5%C3%A5%B7%A5%E7%A5%F3)を確認する。

  - クライアントは700（rwx）
  - サーバ側は600（rw）

- 確認結果：どちらも期待通り。

[Debian](http://d.hatena.ne.jp/keyword/Debian)「心オープンやで〜」

### [Debian](http://d.hatena.ne.jp/keyword/Debian)、お前もしかして...

[Debian](http://d.hatena.ne.jp/keyword/Debian)「さっきから好き勝手疑ってくれてますけど」  
[Debian](http://d.hatena.ne.jp/keyword/Debian)「全然こっちは問題ないですから。」  
[Debian](http://d.hatena.ne.jp/keyword/Debian)「もうひとつ言っておくと、使用する鍵をちゃんと指定すれば[ssh](http://d.hatena.ne.jp/keyword/ssh)接続できますから！こんな感じで。」

```
$ ssh -i ~/.ssh/client_rsa -p 3090 YYYYYYYYYYYY@XXX.XX.XX.XXX
```

## [Debian](http://d.hatena.ne.jp/keyword/Debian)は覚えていた編

### 忘れたのはクライアント（[mac](http://d.hatena.ne.jp/keyword/mac)）のキーチェーン

実は問題があるのは[Debian](http://d.hatena.ne.jp/keyword/Debian)ではなく、キーチェーンの方でした。  
[macOS](http://d.hatena.ne.jp/keyword/macOS) 10.12 [Sierra](http://d.hatena.ne.jp/keyword/Sierra)以降、キーチェーンが再起動時に読み込まれなくなる仕様となったそうです。

タイトルに「突然」とありますが、実際には「[mac](http://d.hatena.ne.jp/keyword/mac)の再起動」をトリガーとして本事象が発生します。（事象発生までは[macbook](http://d.hatena.ne.jp/keyword/macbook)をスリープするだけで、再起動していなかったんでしょうね...）

### キーチェーンに[Debian](http://d.hatena.ne.jp/keyword/Debian)との友情を邪魔させない。

再度`ssh-add -K ~/.ssh/client_rsa`を実行することで、一時的には繋がるようになりますが、再起動すると、また[ssh](http://d.hatena.ne.jp/keyword/ssh)接続できなくなってしまいます。  
そのため、恒久対応として\~/.[ssh](http://d.hatena.ne.jp/keyword/ssh)/configに以下の設定を追加します。

```
Host *  UseKeychain yes  AddKeysToAgent yes
```

### もっと[Debian](http://d.hatena.ne.jp/keyword/Debian)と仲良くなる

せっかくconfigの設定をしたので、毎回ポートやipを打たずに済むような設定もしておきます。

```
Host ikuma-t  HostName XXX.XXX.XX.XX  User ikuma-t  IdentityFile ~/.ssh/client_rsa  Port 3090  TCPKeepAlive yes  IdentitiesOnly yes
```

これで再起動しても`ssh ikuma-t`で繋がるようになりました！
